Hvorfor laget JetBrains Kotlin?

  • Sitat: "Vi ville bli mer produktive ved å bytte til et mer ekspressivt språk."
  • JetBrains brukte altså Java selv, og ønsket bedre produktivitet
  • De ville ha full Java-interop (kalle Java-kode direkte, uten omskrivning)
  • Oppstart i 2011 første stabile versjon kom i 2016
  • Google kunngjorde Kotlin som offisielt Android-språk i 2017 🚀

Det som satte seg

Null-safety

Java

// Kompilerer fint. Kan krasje i prod.
String navn = hentBruker().getNavn().toUpperCase();

Kotlin

// Kompilatoren nekter deg å glemme null
val navn: String? = hentBruker()?.navn
val store = navn?.uppercase() ?: "UKJENT" // ?: Elvis operator: fallback-verdi
Du velger bevisst å håndtere null — ikke glemme det
Det som satte seg

Immutability som default

Java — ingen hindring mot å mutere tilstand

Ordre ordre = new Ordre(Status.MOTTATT);
ordre.setStatus(Status.SENDT);      // lovlig — men var det meningen?
ordre.setStatus(Status.ANNULLERT);  // også lovlig...

Kotlin — val er read-only

data class Ordre(val status: Status)

val ordre = Ordre(Status.MOTTATT)
ordre.status = Status.SENDT           // kompileringsfeil — ikke lov
val sendt = ordre.copy(status = Status.SENDT) // slik gjør man det
data class sin copy(): kopierer alt og lar deg overstyre det du trenger

Smart casts

Java

if (melding instanceof OrdreOpprettet) {
    OrdreOpprettet event = (OrdreOpprettet) melding; // manuell cast
    behandle(event.getOrdreId());
}

Kotlin

if (melding is OrdreOpprettet) {
    behandle(melding.ordreId) // allerede castet — kompilatoren vet
}
"Java-folk kjenner seg kanskje igjen i den manuelle cast-smerten.
Det som satte seg

String templates

Java

String melding = "Hei, " + bruker.getNavn()
    + "! Du har " + antall + " meldinger.";

Kotlin

val melding = "Hei, ${bruker.navn}! Du har $antall meldinger."
Ingen StringBuilder. Ingen String.format. Mer lesbar kode.
Det som satte seg

Default parameters + named arguments

Java — tre overloads for å håndtere defaults

void opprett(String navn) { opprett(navn, "standard", true); }
void opprett(String navn, String type) { opprett(navn, type, true); }
void opprett(String navn, String type, boolean aktiv) { ... }

Kotlin — én metode

fun opprett(navn: String, type: String = "standard", aktiv: Boolean = true) { ... }

opprett("Espen")
opprett("Espen", aktiv = false) // named arguments
Eliminerer overload-pyramider. Named arguments gjør kallsteder selvdokumenterende.

⚠ Java har ingen plan for default parameters eller named arguments — Kotlin har fortsatt et klart forsprang her.

Ikke alt trenger å leve i en klasse

Java — utility-klasse for å få til det samme

public final class OrdreUtils {
    private OrdreUtils() {}
    public static BigDecimal beregnMva(BigDecimal belop) { ... }
}

Kotlin — top-level function

fun beregnMva(belop: BigDecimal): BigDecimal { ... }
I Kotlin er funksjoner førsteklasses borgere, som ikke må eksistere i en klasse

Trailing lambdas

Java — lambda med eksplisitt parameter

List.of(1, 2, 3).stream()
    .filter(n -> n > 1)
    .toList();

Kotlin — lambda utenfor parentesene, én parameter heter automatisk it

listOf(1, 2, 3).filter { it > 1 }
"Første gang jeg så dette: hva er det her?
Andre gang: okei, okei..
Tredje gang: of course!"
Det som satte seg

Extension functions

Java — statisk utility-metode

public class OrgNrUtils {
    public static boolean erGyldig(String orgNr) {
        return orgNr.matches("\\d{9}");
    }
}
OrgNrUtils.erGyldig("974760843"); // true

Kotlin — extension function

fun String.erGyldigOrganisasjonsnummer(): Boolean =
    this.matches(Regex("\\d{9}"))

"974760843".erGyldigOrganisasjonsnummer() // true
Extension functions finnes ikke i Java, og er heller ikke planlagt.

when + sealed classes

Kotlin

sealed interface Status
object Aktiv : Status
object Inaktiv : Status

val tekst = when (status) {
    is Aktiv   -> "Aktiv"
    is Inaktiv -> "Inaktiv"
    // Glemmer du en? Kompileringsfeil!
}
Ny status gir kompileringsfeil alle steder den ikke håndteres
Det som trenger modning

Scope functions

Uten scope functions — vanlig imperativ tilnærming

log.info("Behandler $ordre")
val beregnet = beregn(ordre)
beregnet.valider()
val resultat = lagre(beregnet)

Med scope functions — alt i én kjede

val resultat = ordre
    .also { log.info("Behandler $it") }
    .let { beregn(it) }
    .apply { valider() }
    .run { lagre(this) }
Gjør det samme — men er det mer lesbart? 🤔
Det som trenger modning

Coroutines

Kotlin — suspend functions

// kallEksternApi() kan blokkere — suspend gjør at tråden ikke sitter og venter
suspend fun hentOrdre(id: UUID): Ordre =
    kallEksternApi(id)

Kraftig, men mye nytt å lære.

Java 21+ — Virtual Threads

// kallEksternApi() kan blokkere — Virtual Threads håndterer ventingen for deg
Thread.startVirtualThread(() -> {
    var resultat = kallEksternApi(id);
});
Samme resultat. Ingen nye konsepter å lære.
